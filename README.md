# 比赛计分主机 `score_system`

这是整套系统的主机端，对应裁判客户端项目 [esp32-client](../esp32-client)。

主机负责：
- 创建 WiFi 热点 `ScoreSystem`
- 接收 3 个裁判客户端提交的本轮分数
- 按 2/3 多数规则结算本轮
- 累加总比分
- 提供两个网页界面

## 当前规则

- 每轮只接受 `client1`、`client2`、`client3` 各提交 1 次
- 同一裁判在同一轮重复提交会被拒绝
- 三名裁判全部提交后，本轮自动锁定
- 本轮锁定后，客户端会显示 `----`
- 网页必须点击“下一轮”才会开启下一轮
- 网页点击“重置总比分”时，会同时清空总比分和当前裁判显示分数

## 热点信息

- SSID: `ScoreSystem`
- Password: `12345678`

## 网页地址

- `http://192.168.4.1/`
  - 只显示双方名称和总比分
- `http://192.168.4.1/control`
  - 显示总比分、3 名裁判当前轮分数
  - 提供“下一轮”和“重置总比分”按钮
  - 支持修改双方名称

## 烧录

### 1. 烧录固件

在 PlatformIO 中执行 `Upload`。

### 2. 上传网页文件

网页文件在 `data/index.html` 和 `data/control.html`。

修改网页后必须额外执行一次：
- `Upload Filesystem Image`

如果只烧录固件，不上传文件系统，网页不会更新。

## 页面说明

### 展示页 `/`

- 适合投屏或大屏展示
- 只显示：
  - 队伍名称
  - 总比分

### 控制页 `/control`

- 适合操作人员使用
- 显示：
  - 队伍名称编辑框
  - 总比分
  - 当前轮状态
  - 3 名裁判当前轮分数
  - “下一轮”按钮
  - “重置总比分”按钮

## 轮次状态

- `当前轮次开放中`
  - 裁判客户端可以继续提交
- `当前轮次已锁定，等待开启下一轮`
  - 三名裁判都已提交完成，必须点“下一轮”

## 计分规则

红蓝双方分别独立按以下规则计算：

- 若 3 名裁判中有至少 2 人给出相同分数，则采用该分数
- 若 3 个分数都不同，则取最低分

示例：

| 三位裁判打分 | 最终采用 |
|---|---|
| `5, 5, 7` | `5` |
| `7, 5, 5` | `5` |
| `3, 5, 7` | `3` |

## 接口

| 路径 | 方法 | 说明 |
|---|---|---|
| `/` | `GET` | 返回展示页 |
| `/control` | `GET` | 返回控制页 |
| `/score` | `GET` | 返回总比分、裁判分数、轮次状态 |
| `/roundStatus` | `GET` | 返回当前轮次是否开放 |
| `/updateScore` | `GET` | 裁判客户端提交分数 |
| `/setNames` | `POST` | 设置双方名称 |
| `/nextRound` | `POST` | 开启下一轮 |
| `/resetTotal` | `POST` | 重置总比分和当前轮分数 |

### `/score` 返回示例

```json
{
  "totalA": 12,
  "totalB": 8,
  "roundOpen": false,
  "roundId": 3,
  "judges": [
    { "id": "client1", "scoreA": 5, "scoreB": 3, "submitted": true },
    { "id": "client2", "scoreA": 5, "scoreB": 3, "submitted": true },
    { "id": "client3", "scoreA": 7, "scoreB": 3, "submitted": true }
  ]
}
```

## 故障排查

| 现象 | 说明 |
|---|---|
| 打开网页还是旧内容 | 只烧录了固件，没有执行 `Upload Filesystem Image` |
| 某裁判提交无效 | 同一轮已提交过，或当前轮已锁定 |
| 客户端一直显示 `----` | 网页还没有点“下一轮” |
| 网页打不开 | 确认访问的是 `http://192.168.4.1`，不是 `https://` |

## 已知限制

- 队名、总比分、轮次状态目前只保存在 RAM 中，断电会丢失
- `/updateScore` 没有额外鉴权，只依赖约定的客户端 ID
- 前端仍然是 500ms 轮询，不是 WebSocket 实时推送
